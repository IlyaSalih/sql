# Резервное копирование

## Задание 1
Кейс

Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.

Необходимо описать, какие варианты резервного копирования подходят в случаях:

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

Приведите ответ в свободной форме.

## Решение 

### 1.1. Восстановление данных за предыдущий день
Для этого случая подходит полное резервное копирование (full backup), выполняемое раз в сутки — например, ночью, в период низкой нагрузки. Создаётся полная копия всей базы данных. При сбое достаточно развернуть последнюю суточную копию, и данные восстанавливаются на момент её создания. Такой вариант полностью закрывает требование «восстановить данные за предыдущий день».

### 1.2. Восстановление данных за час до поломки
Одного суточного бэкапа здесь недостаточно, так как между ночной копией и моментом поломки успевают накопиться изменения, которых в копии нет. Чтобы восстановить данные на любой произвольный момент времени, применяется восстановление на точку во времени (Point-In-Time Recovery, PITR). Оно строится на связке полного бэкапа и бинарного лога (binary log), в который непрерывно записываются все изменения базы после снятия копии. Восстановление выполняется в два шага: сначала разворачивается последний полный бэкап, затем поверх него «докатываются» изменения из бинарного лога до нужного момента — за час до аварии. При таком подходе теряются не сутки данных, а лишь секунды или минуты.

### 1.3. Мгновенное переключение на работающую базу
Такой кейс возможен, но решается он не резервным копированием, а средствами обеспечения высокой доступности (High Availability). Резервная копия — это файл, который ещё нужно развернуть, что занимает время и приводит к простою. Мгновенное переключение обеспечивает репликация: рядом с основным сервером (master) работает горячая реплика (slave), которая через репликацию постоянно поддерживает актуальную копию данных. При отказе основного сервера происходит автоматическое переключение нагрузки на реплику (failover) — за секунды, без длительного простоя. Именно так строятся надёжные системы в финансовой сфере.
Важно отметить, что резервное копирование и репликация решают разные задачи и не заменяют друг друга. Бэкап защищает от потери данных, в том числе от логических ошибок (например, случайного удаления — его можно откатить из копии). Репликация защищает от простоя, но не спасает от логических ошибок, так как повреждение данных тут же копируется на реплику. Поэтому в надёжных системах, особенно в финансовых, применяют оба механизма одновременно.

## Задание 2

2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).

2.1.* Возможно ли автоматизировать этот процесс? Если да, то как?

Приведите ответ в свободной форме.

## Решение

### 2.1. Пример команд резервирования и восстановления

Для резервного копирования одной базы данных в PostgreSQL используется утилита pg_dump, а для восстановления — pg_restore или psql, в зависимости от выбранного формата дампа.
Важная особенность: формат резервной копии определяет способ её восстановления. При использовании специального формата (флаг -Fc, custom format) дамп восстанавливается через pg_restore, а при обычном текстовом формате — через psql.

1. Пример резервного копирования в custom-формате:
```
pg_dump -U postgres -Fc mydb > mydb.dump
```
2. Восстановление такой копии (при необходимости предварительно создаётся пустая база):
```
createdb -U postgres mydb
pg_restore -U postgres -d mydb mydb.dump
```
3. 2.1. Автоматизация процесса

Да, процесс резервного копирования можно и нужно автоматизировать. Самый распространённый способ — запуск скрипта с pg_dump по расписанию через планировщик cron. Скрипт снимает дамп, добавляет к имени файла текущую дату и удаляет устаревшие копии, реализуя таким образом ротацию:

```
#!/bin/bash
BACKUP_DIR="/var/backups/postgres"
DATE=$(date +%F_%H-%M)
pg_dump -U postgres -Fc mydb > "$BACKUP_DIR/mydb_$DATE.dump"
find "$BACKUP_DIR" -name "*.dump" -mtime +7 -delete
```

Запуск скрипта каждую ночь настраивается строкой в crontab:

```
0 2 * * * /path/to/backup.sh
```

## Задание 3

3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL.

3.1.* В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?

Приведите ответ в свободной форме.

## Решение

Для организации инкрементного резервного копирования в MySQL обычно используют бинарные логи (binary logs). После создания полного бэкапа можно сохранять только изменения, которые записываются в binlog.
```
mysqlbinlog mysql-bin.000001 > incremental_backup.sql
```
Эта команда позволяет выгрузить изменения из бинарного лога и использовать их для восстановления данных после применения полного резервного копирования.
